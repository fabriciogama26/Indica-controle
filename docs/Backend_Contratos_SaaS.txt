Tela: Backend Contratos SaaS

Visao geral
-----------
- Documento para registrar quais partes do Supabase o frontend web depende hoje.
- Serve para evitar perda de contexto ao separar o SaaS do app Android.

Arquitetura
-----------
- Backend versionado neste repositorio em `supabase/`.
- O contrato minimo atual do SaaS cobre autenticacao web, recuperacao, logout, auditoria e base de permissao por pagina.
- O contrato atual tambem cobre a edicao de credenciais por usuario na tela `/permissoes`.
- O contrato atual tambem cobre a leitura das permissoes salvas por usuario no bootstrap da sessao web.

Cadastro
--------
- Nao se aplica diretamente.

Filtros, auto-complete e listagem
---------------------------------
- Nao se aplica diretamente.

Boas praticas
-------------
- Publicar migrations no Supabase e manter docs atualizadas.
- Nao acoplar frontend web a SQL embutido dentro do Next.js.
- Usar Edge Functions como porta de entrada do frontend quando a regra exigir autenticacao, rate limit e auditoria.

Cancelamento e saldo
--------------------
- Regras de saldo e materiais existem no backend, mas ainda nao foram consumidas pelo frontend web.

Atalhos
-------
- Function de login web: `supabase/edge_functions/auth-login-web/index.ts`
- Function de recover web: `supabase/edge_functions/auth-recover/index.ts`
- Function de logout: `supabase/edge_functions/logout/index.ts`
- Function de log de erro: `supabase/edge_functions/log_error/index.ts`

Mapa de Codigo (funcoes, hooks e constantes)
--------------------------------------------
Funcoes principais
- Caminho: `supabase/edge_functions/auth-login-web/index.ts`
  - Responsavel por: autenticar `login_name + senha`, aplicar rate limit e registrar evento `LOGIN` no `login_audit`.
- Caminho: `supabase/edge_functions/auth-recover/index.ts`
  - Responsavel por: resolver `login_name -> email`, aplicar rate limit e disparar reset de senha.
- Caminho: `supabase/edge_functions/logout/index.ts`
  - Responsavel por: registrar evento `LOGOUT` no `login_audit` usando `session_ref`.
- Caminho: `supabase/edge_functions/log_error/index.ts`
  - Responsavel por: registrar erros do frontend/app.
- Caminho: `src/app/api/app-users/search/route.ts`
  - Responsavel por: listar usuarios reais do tenant atual para a tela `/permissoes`.
- Caminho: `src/app/api/app-users/[userId]/permissions/route.ts`
  - Responsavel por: carregar e salvar `role`, `ativo/inativo` e permissoes por tela do usuario selecionado.
- Caminho: `src/app/api/auth/session-access/route.ts`
  - Responsavel por: resolver o usuario autenticado atual e devolver as telas liberadas em `app_user_page_permissions`.

Constantes e configuracoes
- Caminho: `supabase/migrations/016_add_login_name_to_auth_tables.sql`
  - Responsavel por: adicionar `login_name` ao backend de autenticacao.
- Caminho: `supabase/migrations/017_sync_auth_users_to_app_users.sql`

- Caminho: `supabase/migrations/018_make_auth_user_sync_fail_open.sql`

  - Responsavel por: reforcar as policies RLS para usar `auth.uid()` com `app_users.ativo = true`.
- Caminho: `supabase/migrations/021_rls_to_authenticated.sql`
  - Responsavel por: limitar as policies multi-tenant ao role `authenticated`.
- Caminho: `supabase/migrations/022_create_page_permissions.sql`

  - Responsavel por: criar `app_user_page_permissions` e a funcao `user_has_page_action(text, text)` para a matriz por usuario e por tela.

  - Responsavel por: liberar leitura de `app_users` do mesmo tenant apenas para perfis administrativos autenticados.
- Caminho: `supabase/migrations/011_create_rate_limit.sql`
  - Responsavel por: rate limit das functions.

  - `021_rls_to_authenticated.sql`
  - `022_create_page_permissions.sql`
  - `023_normalize_roles_to_app_roles.sql`
- Functions minimas para o SaaS web:
  - `auth-login-web`
  - `auth-recover`
  - `logout`
  - `log_error`
- Nenhuma RPC nova e obrigatoria para login, recover ou shell nesta fase alem da `rate_limit_check_and_hit`.
- Criar usuario apenas em `Authentication > Users` nao deve mais falhar por causa da sincronizacao. O vinculo com `app_users` ainda depende de email unico correspondente ou metadata minima no Auth (`tenant_id` e `matricula`).
- `login_audit` agora registra duas linhas por sessao do SaaS web: uma com `event_type = 'LOGIN'` e outra com `event_type = 'LOGOUT'`, correlacionadas por `session_ref`.
- As policies RLS multi-tenant agora devem considerar `auth.uid()` e `app_users.ativo = true` para evitar acesso de usuarios desativados com sessao ainda valida.
- As policies multi-tenant agora tambem devem ser `TO authenticated`, evitando depender apenas de `auth.uid() is null` para bloquear `anon`.
- `app_pages` funciona como catalogo global de rotas controladas e `role_page_permissions` armazena acesso por tenant e `role_id`.
- `app_roles` virou a fonte de verdade dos perfis (`admin`, `master`, `user`).
- A funcao `user_is_admin_in_tenant(uuid)` agora resolve o papel administrativo via `app_roles.is_admin`.
- A busca da tela `/permissoes` valida o operador atual como admin/master e consulta `app_users` do mesmo tenant com `service_role` no backend, sem expor esse acesso ao frontend.
- O operador da tela `/permissoes` precisa ser admin/master, mas a busca retorna qualquer `app_user` do mesmo tenant para configuracao de acesso.
- O salvamento da tela `/permissoes` nao usa `delete`; ele faz `upsert` em `app_user_page_permissions` e espelha o mesmo toggle visual em `can_access`, `can_select`, `can_insert` e `can_update`.
- O shell e a protecao de rotas nao devem confiar so em `role`; quando `hasCustomPermissions = true`, a navegacao passa a ser definida pelas linhas salvas em `app_user_page_permissions`.
