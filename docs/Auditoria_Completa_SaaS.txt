Auditoria Completa - Indica Controle SaaS
==========================================
Data: 2026-03-22
Escopo: revisao de integridade de dados, tabelas Supabase, duplicidade de codigo, edge functions, padronizacao de telas e documentacao.

================================================================================
1. RESUMO EXECUTIVO
================================================================================
O projeto possui uma base solida de seguranca multi-tenant (RLS, RPCs transacionais, validacao de concorrencia com expectedUpdatedAt). Porem, existem problemas de manutenibilidade graves:
- Duplicidade de funcoes utilitarias em ate 7 arquivos.
- Views monoliticas com 3500+ linhas e 128KB em um unico arquivo.
- Edge functions copy-paste com ~95% de codigo identico.
- 14 de 27 telas sem documentacao TXT.
- Fallback progressivo de queries com 4 tentativas em cascata (tecnica de compatibilidade que pode mascarar erros).

================================================================================
2. DUPLICIDADE DE CODIGO (CRITICO)
================================================================================

2.1. Funcoes utilitarias duplicadas no frontend
------------------------------------------------
As seguintes funcoes aparecem copiadas em multiplos arquivos PageView.tsx:

| Funcao                  | Ocorrencias | Arquivos                                               |
|-------------------------|-------------|-------------------------------------------------------|
| formatDateTime          | 7           | Projetos, ProgramacaoSimples, Pessoas, Materiais, Locacao, Equipes, Atividades |
| downloadCsvFile         | 6           | Projetos, ProgramacaoSimples, Pessoas, Materiais, Equipes, Atividades          |
| toIsoDate               | 3           | ProgramacaoSimples, Programacao, Medicao                |
| formatDate              | 3           | Projetos, ProgramacaoSimples, Medicao                   |
| parseNonNegativeInteger | 2           | ProgramacaoSimples, Locacao                             |

Recomendacao:
- Criar `src/lib/utils/formatters.ts` com: formatDate, formatDateTime, toIsoDate, formatDateShort.
- Criar `src/lib/utils/csv.ts` com: downloadCsvFile.
- Criar `src/lib/utils/parsers.ts` com: parseNonNegativeInteger, parseOptionalPositiveInteger, parsePositiveNumber.
- Importar de cada PageView ao inves de copiar.
- Impacto: reduz ~200 linhas duplicadas e garante correcao uniforme.

2.2. Edge Functions quase identicas
------------------------------------
- `import_project_forecast/index.ts` (333 linhas)
- `import_project_activity_forecast/index.ts` (334 linhas)

Ambas compartilham ~95% do codigo:
- corsHeaders, respond, getBearerToken, normalizeText, normalizeHeader, parsePositiveNumber, parseWorkbook (identicas).
- Autenticacao, validacao de projeto, leitura de formulario (identicas).
- Unica diferenca: tabela de lookup (materials vs service_activities), RPC de precheck e RPC de append.

Recomendacao:
- Criar um modulo compartilhado `supabase/edge_functions/_shared/xlsx-import-utils.ts` com as funcoes comuns.
- Cada edge function importa o modulo e configura apenas a tabela, campo de lookup e RPCs.
- Impacto: reduz ~270 linhas duplicadas entre os dois arquivos.

3. VIEWS MONOLITICAS (ALTO RISCO DE MANUTENCAO)
================================================================================

| Arquivo                        | Linhas | Tamanho |
|-------------------------------|--------|---------|
| ProgrammingSimplePageView.tsx | 3566   | 128 KB  |
| ProjectsPageView.tsx          | ~3500  | 129 KB  |
| ProgrammingPageView.tsx       | ~2400  | 84 KB   |
| LocationPageView.tsx          | ~2200  | 78 KB   |

Problemas:
- Todos os types, utils, constantes e estado inline em um unico componente.
- Dificulta code review, testes e reutilizacao.
- Hot reload lento em desenvolvimento.

Recomendacao para padrão novo de telas:
- Separar types em `types.ts` no mesmo modulo.
- Separar utilitarios em `utils.ts` ou importar de `src/lib/utils/`.
- Separar constantes em `constants.ts`.
- Manter o componente PageView focado em rendering e orquestracao de estado.
- Meta: PageView com no maximo ~800-1000 linhas.

================================================================================
4. INTEGRIDADE DE DADOS E SALVAMENTO
================================================================================

4.1. Pontos positivos (ja bem feitos)
--------------------------------------
- Concorrencia: `expectedUpdatedAt` exigido em PATCH para todas as telas operacionais (Programacao, Projetos, Locacao, Pessoas, Materiais, Equipes, Atividades).
- Conflito rico: resposta 409 com `currentRecord`, `currentUpdatedAt`, `updatedBy`, `changedFields`.
- RPCs transacionais: `save_project_programming_full` e `save_project_programming_batch_full` fazem rollback total em caso de erro.
- Historico transacional: o historico de CREATE/UPDATE/RESCHEDULE/BATCH_CREATE e gravado dentro da mesma transacao do save (migration 106).
- Adiamento protegido: valida data futura no frontend, API e RPC.
- Importacao em massa: precheck de duplicata antes do insert (edge functions verificam codigo duplicado no arquivo e no banco).

4.2. Riscos e melhorias necessarias
------------------------------------
a) Fallback progressivo na API de Programacao (route.ts, linhas 716-786):
   - `fetchProgrammingRows` tenta 4 SELECTs diferentes em cascata, do mais completo ao legado.
   - Risco: erros no SELECT mais completo (ex.: timeout) podem silenciosamente degradar para um SELECT antigo sem alertar o usuario.
   - Recomendacao: se todas as migrations ja estao aplicadas, remover os fallbacks e usar apenas o SELECT completo. Se compatibilidade e necessaria, logar quando um fallback e ativado.

b) Ausencia de `useErrorLogger`:
   - O AGENTS.md menciona `useErrorLogger('<modulo>')` como padrao para erros de tela.
   - Nenhum modulo implementa isso atualmente (0 ocorrencias no codigo).
   - Recomendacao: criar o hook `useErrorLogger` e integrar nas telas operacionais.

c) Telas de cadastro-base sem transacao:
   - Modulos como `cadastro-base`, `cargo`, `centro-servico`, `contrato`, `municipio`, `nivel-tensao`, `porte`, `prioridade`, `responsavel-distribuidora`, `tipo-equipe`, `tipo-servico` sao telas de lookup simples.
   - Verificar se o CRUD dessas telas passa por RPC ou faz INSERT direto via Supabase client, o que poderia bypassar validacao.

d) Medicao (MeasurementPageView.tsx):
   - Tela de medicao parece menos madura que Programacao. Verificar se possui controle de concorrencia igual.

9. PADRAO DE TELA RECOMENDADO (PARA NOVAS TELAS)
================================================================================

Baseado na analise da Programacao Simples, o padrao ideal para novas telas:

Estrutura de arquivos:
  src/modules/dashboard/<nome-tela>/
    types.ts              - tipos (FormState, FilterState, *Item, *Response)
    constants.ts          - constantes (PAGE_SIZE, FIELD_LABELS, DOCUMENT_KEYS)
    utils.ts              - utilitarios especificos da tela (funcoes de normalizacao local)
    <NomeTela>PageView.tsx         - componente principal
    <NomeTela>PageView.module.css  - estilos

Imports compartilhados (criar se nao existir):
  src/lib/utils/formatters.ts  - formatDate, formatDateTime, toIsoDate
  src/lib/utils/csv.ts         - downloadCsvFile
  src/lib/utils/parsers.ts     - parseNonNegativeInteger, parsePositiveNumber

Salvamento:
  - SEMPRE usar RPC transacional (save_*_full) para operacoes com efeitos colaterais.
  - SEMPRE exigir expectedUpdatedAt no PATCH/PUT para controle de concorrencia.
  - SEMPRE retornar o registro salvo no response para evitar "salvar e buscar novamente".
  - Em lote: atomico com rollback total.
  - Historico dentro da transacao.

Concorrencia:
  - 409 Conflict com: currentRecord, currentUpdatedAt, updatedBy, changedFields.
  - UX de conflito com mensagem objetiva e acao para recarregar.

Feedback:
  - Sucesso com aviso quando registro salvo esta fora dos filtros.
  - Modal de alerta para erros de validacao (nao apenas mensagem no topo).
  - Scroll ao topo apos sucesso/erro.

Exportacao:
  - Cooldown de 10 segundos entre exportacoes.
  - CSV + layout dedicado quando aplicavel.

Auditoria:
  - Historico operacional em tabela propria (*_history).
  - Modal de historico com paginacao.
  - useErrorLogger('<modulo>') para erros de tela (IMPLEMENTAR).
